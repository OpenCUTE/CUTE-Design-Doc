# Phase 3 — 软件与测试

## 方法论

本阶段采用 **测试金字塔（Test Pyramid）** 与 **知识迁移（Knowledge Transfer）** 方法论：

- **测试金字塔（Test Pyramid）**：CUTE 的 5 层测试架构天然呈现金字塔结构 — 底层是大量快速的单元测试，向上逐层递减到集成测试和端到端测试。文档需要清晰描述每一层的定位、编写方法和运行方式，使开发者能在正确的层级编写正确的测试。
- **知识迁移（Knowledge Transfer）**：CUTE 已有 355 个测试文件和丰富的测试实践，但相关知识可能分散在代码注释、PR 描述和开发者脑中。本阶段的文档工作是将这些隐性知识显性化、结构化，降低新贡献者的上手门槛。
- **以用例驱动（Use-Case-Driven）**：测试框架文档以"我想测什么 → 应该用哪一层 → 怎么写"的用例路径组织，而不是按框架的 API 列表组织。
- **可复现性（Reproducibility）**：基准测试结果必须包含完整的环境信息、配置参数和复现步骤，确保结果可验证。

       L5 --> L4 --> L3 --> L2 --> L1
   ```

## 2. 测试层次详解

### Layer 1: 单元测试（Unit Tests）
   - **定位：** 验证单个硬件组件的基本功能
   - **范围：** 单个 Chisel 模块（如单个 PE、单个 Scratchpad）
   - **运行方式：** ChiselTest / treadle
   - **示例：** 验证一个 PE 的 MAC 运算正确性
   - **文件组织：** `src/test/scala/.../Unit/`
   - **数量估计：** 最多（占测试文件的大多数）

### Layer 2: 模块功能测试（Module Tests）
   - **定位：** 验证单个子系统的完整功能
   - **范围：** 完整的子系统（如 MTE 阵列、Scratchpad 组）
   - **运行方式：** ChiselTest + Verilator 后端
   - **示例：** 验证 MTE 4×4 阵列的矩阵乘法
   - **文件组织：** `src/test/scala/.../Module/`

### Layer 3: 模块交互测试（Integration Tests）
   - **定位：** 验证多个子系统之间的协作
   - **范围：** 计算引擎 + 存储系统 + 控制逻辑
   - **运行方式：** ChiselTest + Verilator 后端
   - **示例：** 数据从 Scratchpad 加载到 MTE 再写回的完整流程
   - **文件组织：** `src/test/scala/.../Integration/`

### Layer 4: 系统集成测试（System Tests）
   - **定位：** 验证 CUTE 在 CPU 系统中的正确集成
   - **范围：** CPU 核 + RoCC 接口 + CUTE + 主存
   - **运行方式：** Rocket Chip 仿真 + DRAMSim3
   - **示例：** 通过自定义指令触发矩阵运算并验证结果
   - **文件组织：** `src/test/scala/.../System/`

### Layer 5: 端到端测试（End-to-End Tests）
   - **定位：** 验证完整应用场景的正确性
   - **范围：** 从软件 API 到硬件执行的完整链路
   - **运行方式：** 完整系统仿真
   - **示例：** 执行一个完整的矩阵乘法 benchmark
   - **文件组织：** `src/test/scala/.../E2E/`

## 3. 测试基础设施

### 3.1 测试运行器
   - 使用的测试框架（ChiselTest / UTTest / 其他）
   - SBT 测试配置
   - 并行测试策略

### 3.2 测试数据生成
   - 随机数据生成策略
   - 参考模型（Golden Model）
   - 数据类型覆盖策略

### 3.3 参考模型
   - 软件参考模型的实现方式
   - 精度匹配策略（浮点比较容差）
   - 不同数据类型的参考模型

## 4. CI 中的测试
   - GitHub Actions 中的测试配置
   - 测试分层运行策略（PR 只跑 L1-L3，merge 跑全部）
   - 测试超时和资源限制
